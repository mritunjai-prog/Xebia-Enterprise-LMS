## PART 2: DATABASE DESIGN

### New Entity: `TrainerAllocation` (batch-service)

**Table**: `trainer_allocations`

| Column | Type | Constraints | Purpose |
|---|---|---|---|
| id | String (UUID) | PK, auto-generated | Unique allocation ID |
| trainer_id | String | NOT NULL, indexed | References user-service User.id |
| batch_id | String | NOT NULL, indexed | References batch-service Batch.id |
| course_id | String | NOT NULL, indexed | References course-service Course.id (logical FK) |
| academic_session | String | NOT NULL | e.g., "2023-2027", "2024-2028" |
| status | String | NOT NULL, default 'active' | active, completed, cancelled |
| start_date | String | | ISO date |
| end_date | String | | ISO date |
| assigned_by | String | | Admin user ID who made the allocation |
| assigned_at | String | NOT NULL | ISO datetime |
| notes | String | | Optional notes |

**Indexes**: (trainer_id), (batch_id), (course_id), (trainer_id, batch_id), (academic_session)

**Why this design**: The `trainer_allocations` bridge table supports ALL allocation rules:
- One Trainer → Multiple Batches: query by trainer_id
- One Trainer → Multiple Courses: query by trainer_id, group by course_id
- One Course → Multiple Trainers: query by course_id, group by trainer_id
- Same Batch → Multiple Courses: query by batch_id, group by course_id

**Note**: University field removed — not needed for allocation. Batches are selected directly by name/ID.

### Modified Entity: `Batch` (batch-service)

**New fields to add**:

| Column | Type | Purpose |
|---|---|---|
| trainer_id | String | Current assigned trainer (denormalized from TrainerAllocation for quick access) |
| course_id | String | Proper FK reference to course ID (replaces free-text course) |
| academic_session | String | Academic session |
| start_date | String | Batch start date |
| end_date | String | Batch end date |
| max_students | Integer | Maximum student capacity |
| created_by | String | Trainer ID who created this batch (set at creation time) |
| created_by_name | String | Trainer name who created this batch (denormalized for display) |

**Why `created_by` / `created_by_name`**: These fields track which trainer created the batch in the trainer portal. The Allocation Wizard uses this to auto-determine the trainer when a batch is selected, eliminating the need for a separate university step.

**Note**: `university` field removed — not needed. Batches are identified by name and creator, not by university.

### Modified Entity: `User` (user-service)

**New fields to add**:

| Column | Type | Purpose |
|---|---|---|
| specialization | String | Trainer's primary domain |
| max_batches | Integer | Maximum concurrent batch assignments (default 5) |

**Note**: No university field needed — trainers are associated with batches via the batch's `created_by` field.

## PART 8: ALLOCATION WIZARD — DETAILED CHANGE SPEC

### Problem
The current Allocation Wizard has 5 steps including a University selection step with hardcoded fallback data. The user wants to remove this and show batches created by trainers in the trainer portal directly.

### New Wizard Flow (4 steps)

**Step 1: Select Batch**
- Show all batches fetched from `BatchService.getBatches()`
- Each batch card shows: batch icon, batch name, course focus, creator name (`created_by_name`), student count
- Search filter by batch name
- Click to select a batch
- No university filtering — all batches are visible

**Step 2: Select Trainer**
- Auto-populate with the trainer from `selectedBatch.created_by`
- Show trainer card with avatar initial, name, email
- Admin can override by selecting a different trainer from the full list
- If `created_by` is null (old batches), show all trainers for manual selection

**Step 3: Select Course**
- Multi-select from admin-created courses
- Search filter by course title
- Each card shows checkbox, title, difficulty level, language

**Step 4: Review & Assign**
- Left: Allocation summary (batch, trainer, courses)
- Right: Schedule form (academicSession, startDate, endDate, notes)
- Below: Selected courses list with remove buttons
- Submit creates bulk allocations

### Computation Logic

**Dashboard KPIs**:
- `totalAssessments` = assessmentRepository.count()
- `activeAssessments` = startDate <= now <= endDate AND status == "published"
- `scheduledAssessments` = startDate > now AND status == "published"
- `completedAssessments` = endDate < now AND status == "published"
- `averageScore` = AVG(submission.percentage) WHERE isEvaluated == true
- `averagePassRate` = (count WHERE percentage >= passingMarks) / totalEvaluated * 100
- `studentsAttempted` = COUNT(DISTINCT submission.studentId) WHERE status == "submitted"

**Student Report**:
- Get all students from batch.students for the assessment's assigned batches
- Get all submissions for the assessment
- Left-join: every assigned student appears even if no submission
- Students with submissions: compute score, percentage, pass/fail, timeTaken
- Students without submissions: status = "Not Attempted"
- Students who started but didn't submit: status = "Absent"
