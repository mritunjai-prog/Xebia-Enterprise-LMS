3.2 HikariCP Connection Pool Tuning

Before testing, the database connection pool was tuned in each service's application.yml to prevent connection exhaustion under high load: 

# Applied to all 5 microservice application.yml files
spring:
  datasource:
    hikari:
      maximum-pool-size: 10        # 5 services × 10 = 50 max connections
      connection-timeout: 30000    # 30 second wait before fail
ℹ️
Why pool size 10? Render's free-tier PostgreSQL enforces a hard cap of 50 total concurrent connections. With 5 microservices, setting the pool to 10 per service ensures that even under full burst, the system never exceeds the physical limit: 5 × 10 = 50 connections total.

4 Test Methodology

Unlike sequential tools like Postman Collection Runner (which send requests one after another), this test was written in Node.js using Promise.all() to fire all 50 requests in true parallel — simulating the worst-case real-world scenario of an entire classroom hitting "Submit" at the exact same millisecond.

4.1 The Test Harness (Node.js)

const CONCURRENCY = 50;

// Fire 50 requests simultaneously using Promise.all
const requests = Array.from({ length: CONCURRENCY }).map(async (_, i) => {
  try {
    const res = await fetch(service.url, {
      method: 'POST',
      headers,
      body: JSON.stringify(service.getPayload(i))
    });
    if (!res.ok) throw new Error(await res.text());
    successes++;
  } catch (err) {
    errors.push(err.message);
  }
});

await Promise.all(requests); // TRUE parallel execution

4.3 Pre-Test Database Cleanup

Before each formal test run, a database cleanup script was executed to wipe all records from previous test runs, ensuring a clean baseline: 

// clear_databases.js — Sequential cleanup to avoid connection exhaustion
async function clearService(service) {
  const res = await fetch(service.getAll);
  const records = await res.json();
  for (const record of records) {
    await fetch(`${service.delete}/${record.id}`, { method: 'DELETE', headers });
  }
}
// Result: Deleted 487 stale records across all 5 databases

6 Errors Encountered & Root Causes

During the testing process, several errors were encountered across different iterations. Each error was diagnosed, root-caused, and resolved. This section documents the complete error history.

ERROR #1 — Initial Test Run
HTTP 500 — Database Connection Pool Exhausted
Affected: All 5 services during concurrent GET phase
Root Cause: Firing 50 simultaneous findAll() full-table scans consumed all 50 of Render's free-tier PostgreSQL connections instantly, causing subsequent requests to time out waiting for a free connection. 
// Error response received from all services during GET phase:
HTTP 500 - "could not execute statement"
// Render DB max connection limit: 50 total concurrent connections
ERROR #2 — User Service POST
HTTP 500 — Unique Constraint Violation (email & xebiaId)
Affected: User Service POST
Root Cause: Test payloads used hardcoded sequential integers (user1@xebia.com, user2@xebia.com). After the first test run, the database still had those records. The second run tried to insert the same emails, which violated the UNIQUE constraint. 
HTTP 500: "Server error: could not execute statement"
// DB constraint: UNIQUE(email), UNIQUE(xebiaId)
ERROR #3 — Event Service POST (Round 1)
HTTP 500 — Missing Required Fields
Affected: Event Service POST
Root Cause: The initial Event Service payload was only sending active: true and createdBy, missing the five required fields: title, startDateTime, endDateTime, registrationDeadline, and description. 
// Incorrect payload (missing 5 required fields):
{ "active": true, "createdBy": "..." }

HTTP 500: Internal Server Error
// JPA attempted to persist null values to NOT NULL columns
ERROR #4 — Event Service POST (Round 2)
HTTP 400 — Business Validation Failure (registrationDeadline after startDateTime)
Affected: Event Service POST (all 50 requests)
Root Cause: A strict business rule in EventService.java enforces that the registration deadline must be strictly before the event's start time. The test payload set registrationDeadline to two days after the event, which the API correctly rejected. 
// EventService.java — Business validation logic:
if (req.getRegistrationDeadline().isAfter(req.getStartDateTime())) {
  throw new ResponseStatusException(
    HttpStatus.BAD_REQUEST,
    "Registration deadline must be before event start"
  );
}

// Test sent registrationDeadline = startDateTime + 2 days → REJECTED
HTTP 400: Bad Request
ERROR #5 — Event Service POST (Round 3)
HTTP 400 — Past startDateTime Rejected
Affected: Event Service POST
Root Cause: The test script set startDateTime to the exact current millisecond. By the time the HTTP request traversed the internet to Render's servers (~100-300ms), the timestamp was already technically in the past, failing the server's temporal validation.

7 Fixes Applied

FIX #1 — Database Connection Exhaustion
HikariCP Pool Size Tuned; Test Strategy Revised
Fix: Configured HikariCP's maximum-pool-size: 10 in all 5 service application.yml files. Also revised the GET test strategy from a blind full-table scan to using IDs captured from the POST phase — eliminating unnecessary full-table reads. 
# Fix applied to all 5 application.yml files:
spring.datasource.hikari:
  maximum-pool-size: 10
  connection-timeout: 30000
FIX #2 — Unique Constraint Violation
Timestamp-based Unique Identifier Generation
Fix: Replaced hardcoded sequential indices with Date.now() + i to ensure every test run across all 50 concurrent requests generates globally unique email addresses and Xebia IDs — even when running multiple test rounds on a populated database. 
// Before (broken — causes UNIQUE constraint violations):
email: `testuser${i}@xebia.com`

// After (correct — globally unique across all test runs):
const unique = Date.now() + i;
email: `testuser${unique}@xebia.com`  // e.g. testuser1752706451382@xebia.com
FIX #3 — Event Service Missing Fields
Full Payload Constructed from Schema Inspection
Fix: Inspected the Event.java entity class to identify all @Column(nullable = false) fields and added them all to the test payload. 
// Event.java entity — @Column(nullable = false) fields identified:
@Column(nullable = false) private String title;
@Column(nullable = false) private Instant startDateTime;
@Column(nullable = false) private Instant endDateTime;
@Column(nullable = false) private Instant registrationDeadline;
@Column(nullable = false) private String createdBy;
FIX #4 — Event Service Business Logic Violation
Registration Deadline Set Before Event Start
Fix: Corrected the date ordering: registrationDeadline = now + 1 hour, startDateTime = tomorrow, endDateTime = tomorrow + 1 hour. This respects the API's business constraint and also avoids the past-date transmission issue. 
// Corrected date ordering — satisfies all business rules:
registrationDeadline: new Date(Date.now() + 3600000).toISOString(),   // +1 hour
startDateTime:       new Date(Date.now() + 86400000).toISOString(),  // +24 hours
endDateTime:         new Date(Date.now() + 90000000).toISOString(),  // +25 hours
// Result: registrationDeadline < startDateTime ✅
