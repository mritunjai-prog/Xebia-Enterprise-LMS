1. Introduction and Platform Vision

The Xebia Enterprise Learning Management System (LMS) Portal is a proprietary, modular, high-performance web platform for internal learning, skill acquisition, and certification. It provides a dual-portal experience:

Admin Control Centre

Secure CMS for curriculum architects to build learning hierarchies, upload multimedia, manage courses, and monitor platform-wide analytics. Accessible at /admin.

Student / Learner Portal

Immersive content consumption environment with progress tracking, assessments, and capability badges. Accessible at /student.

Architecture Constraint
Strictly decoupled Client-Server architecture. React frontend and Java backend are independently deployable, allowing UI or database replacement without systemic friction.

4. System Architecture

Three-tier decoupled architecture: React SPA frontend, Spring Cloud API Gateway, Java microservice backend, PostgreSQL database.

5. Application Startup Sequence

5.1 Frontend

  1. start.js bootstraps the React app via Vinxi SSR entry point.
  2. router.js creates the TanStack Router instance and wires up the auto-generated route tree with QueryClient context.
  3. __root.jsx renders the HTML shell: injects dark-mode detection script (reads localStorage.lms_theme), critical anti-FOUC CSS, and Google Fonts links.
  4. The router matches the current URL and renders the appropriate layout and page component tree.

5.2 Backend

  1. Start PostgreSQL on port 5432 (locally or via Docker).
  2. Start course-service — Hibernate auto-creates or migrates database tables.
  3. Start api-gateway on port 8080.
  4. Start frontend dev server: npm run dev on port 3000.

6. API Request Lifecycle

All API calls use fetchApi(endpoint, options) in src/services/api.js. This wrapper:

  • Constructs full URL from VITE_API_BASE_URL.
  • Injects Content-Type: application/json, X-Tenant-Id, X-User-Id on every request.
  • Throws on non-2xx responses, parsing the backend JSON error message.
  • Returns null for 204 No Content responses.

Path: Component calls service function → fetchApi() → API Gateway (port 8080) → Course Service (port 8081) → PostgreSQL → JSON response.

25. File Upload Engine (Cloudinary)

File: src/lib/cloudinary.js. Uses unsigned direct browser uploads. No server-side upload proxy is needed.

Upload Flow

  1. User selects file in CreateCategory or CreateCourse.
  2. handleImageUpload(file) creates a FormData object with the file, upload_preset=lms_uploads, and optional folder.
  3. POSTs directly to https://api.cloudinary.com/v1_1/{VITE_CLOUDINARY_CLOUD_NAME}/upload.
  4. Cloudinary returns a JSON response containing a secure_url field.
  5. getCloudinaryAssetUrl(result) extracts the URL and stores it in the form field.

Utility Functions

FunctionPurpose
isDocumentUrl(url)Returns true if URL points to PDF, DOC, PPT, XLS, etc.
getCloudinaryAssetUrl(uploadResult)Extracts the secure_url from a Cloudinary upload API response.
normalizeCloudinaryDocumentUrl(url)Converts /raw/upload/ document URL to /image/upload/ for browser rendering.
getCloudinaryDocumentPreviewUrl(url)Generates a JPG preview URL for the first page of a PDF using Cloudinary pg_1,f_jpg transformation.

26. Feature Workflows

26.1 Category Creation

  1. Admin clicks Create Category button. showCreatePage becomes true and the CreateCategory component replaces the list.
  2. Admin enters the Name. Optionally clicks AI Generate which calls aiService.generateCategoryDescription(name) via Groq to fill the description field automatically.
  3. Admin selects a colour from the preset brand palette or the custom colour picker.
  4. Admin uploads an icon image. Sent directly to Cloudinary via FormData POST. The returned secure_url is stored in the icon field.
  5. Admin clicks Save. handleSubmit() validates the form and calls CategoryService.createCategory(data) — POST /api/categories.
  6. On success: success toast fires, onBack() navigates back to the list, which re-fetches to show the new category.

26.2 Course Creation

  1. Admin opens CreateCourse form with 5 tabs.
  2. Basic Details: Title (required), Category (required), Description (optional AI-generate), Level, Duration.
  3. Learning Content: Learning Outcomes, Prerequisites, Highlights — all support AI generation via Groq.
  4. Media: Thumbnail and Banner uploaded to Cloudinary via unsigned preset.
  5. Advanced: Published and Active toggles. Slug is auto-generated from the title.
  6. Admin clicks Save. Calls POST /api/courses. On success shows toast.

26.3 Curriculum Building

  1. Admin opens Course Detail and clicks the Curriculum tab. HierarchyBuilder renders.
  2. Admin clicks Add Module. A form appears for the module title. On save: POST /api/courses/:id/modules.
  3. Inside a module, admin clicks Add SubModule. On save: POST /api/courses/:id/modules/:moduleId/submodules.
  4. Inside a submodule, admin clicks Add Content. ContentManager opens. Admin selects type (VIDEO, PDF, TEXT, LINK, QUIZ), provides title and URL or body.
  5. On save: POST /api/courses/:id/content-items. The hierarchy tree re-fetches and displays the updated structure.

26.4 Student Learning

  1. Student navigates to /student/courses and selects a course.
  2. Course Player loads the full hierarchy via getCourseHierarchy().
  3. Student clicks a content item. The appropriate viewer renders: video player, PDF embed, or text block.
  4. ProgressService.updateAccess(courseId, submoduleId, contentId) records the content access event.
  5. Student clicks Mark Complete. ProgressService.markComplete(courseId, submoduleId) is called. Progress bar updates optimistically in the UI.

28. Deployment and DevOps

28.1 Local Development

Start PostgreSQL
docker compose up -d postgres

Start backend services
cd backend/api-gateway
mvn spring-boot:run

cd backend/course-service
mvn spring-boot:run

28.2 Production Deployment

  • Frontend: Build with npm run build. Deploy the dist folder to Vercel, Netlify, Lovable, or any CDN.
  • Backend: Build with mvn package. Deploy the .jar files on a cloud VM, Kubernetes cluster, or container service.
  • Environment Variables: Set in the hosting platform dashboard. Never commit to source control.
  • Cloudinary: Ensure the lms_uploads upload preset is set to Unsigned in the Cloudinary dashboard settings.

28.3 Required Frontend Environment Variables

VariablePurpose

Security Note
VITE_ environment variables are embedded in the JavaScript bundle and are visible to end users. Never place database passwords or private backend secrets in frontend environment variables. Configure Cloudinary upload preset rate limits and Groq API usage caps as safeguards in their respective dashboards.
