This document was generated by analyzing the actual source code in src (frontend) and backend/course-service/src/main/java/com/xebia/lms/course (backend). It reflects the codebase state after the Phase 4 architectural simplification.

0. System-Wide Notes (read this first)

  • Frontend Architecture: The application is a Single Page Application built on React 19 and Vite. Routing is handled by Tanstack Router (@tanstack/react-router) for type-safe routing, defined in src/routes/.
  • State Management: Zustand is utilized for global state management (src/admin/store/useAppStore.js), which handles sidebar toggling, active tenant, toast notifications, modal states, and admin profile data. Local component state handles most form inputs.
  • Backend Architecture: A robust Spring Boot microservices architecture containing Course Service, Common Lib, API Gateway, and Eureka Service Registry. It relies on a PostgreSQL database and MapStruct for DTO mappings. Note that the Course Service exposes its endpoints under /courses, /categories, etc.
  • Mock API & Services: Some services in the frontend currently act as a mock backend for demonstration and development purposes, leveraging localStorage to simulate database persistence (e.g., src/admin/services/api.js has mocked data for dashboard analytics, users, and organizations).
  • UI Library: The frontend leverages lucide-react for iconography, framer-motion for smooth animations, and a bespoke Tailwind CSS configuration integrated with shadcn/ui Radix primitives.

Feature 1: Dashboard Analytics

1. Feature Name

Dashboard Overview & Analytics

2. Feature Purpose

Provides LMS administrators with a high-level overview of key performance indicators (KPIs) like total published/draft courses, active categories, recent course creations, and quick actions to streamline their workflow.

3. Feature Location

Frontend - Component: src/admin/pages/Dashboard/index.jsx - Route Configuration: src/routes/index.jsx - Mock API Data: src/admin/services/api.js

4. Related Files

Frontend - ✓ src/admin/pages/Dashboard/index.jsx - ✓ src/routes/index.jsx - ✓ src/admin/services/api.js

5. Feature Entry Point

User navigates to `/` (Home)
        ↓
Tanstack Router loads `Route` from `src/routes/index.jsx`
        ↓
Renders `Dashboard` component (`src/admin/pages/Dashboard/index.jsx`)

6. Complete Feature Lifecycle

On mount, the Dashboard component loads mock KPI statistics and recent organizations/courses from localStorage via api.dashboard.getOverview(). It sets the local state for stats and recentCourses. The UI then displays a 4-column KPI grid, a list of recently created courses with difficulty badges, and a "Quick Creator Actions" panel on the side.

7. API Details

Currently simulated in the frontend via api.dashboard.getOverview() which resolves a promise after a mock delay, returning mockDashboardData.

8. Database Flow

Uses mocked JSON stored in the browser's localStorage under the key lms_mock_dashboard.

9. State Management

React local state (useState) manages the loading indicator and the stats / recentCourses objects.

10. Component Hierarchy

<Dashboard>
   ├── <Header / Welcome Message>
   ├── <KPI Cards Grid> (4 items mapped from stats)
   ├── <Recent Courses List>
   └── <Quick Actions Panel>

11. Sequence Diagram (Text)

1. Router loads Dashboard
2. Dashboard calls api.dashboard.getOverview()
3. api.js loads 'lms_mock_dashboard' from localStorage
4. Promise resolves with data
5. Dashboard sets stats and recentCourses state
6. Component re-renders with KPI data and course list

12. Dependencies

lucide-react (Icons), @tanstack/react-router (Links), clsx (Class merging).

13. Validation Rules

None active. Read-only view.

14. Error Handling

A generic try-catch block logs errors to the console, but the mock service guarantees a return unless the localStorage parse fails gracefully.

15. Security

Open access. Protected visually via app shell boundaries, but lacks strict route guards or RBAC logic.

16. Important Classes and Methods

api.dashboard.getOverview()

17. Feature Dependencies

Relies on the Course Management and Category Management mock creation flows to populate the `recentCourses` and `stats` if integrated with real endpoints.

18. Possible Improvements

Connect to a real Spring Boot metrics endpoint or analytics microservice instead of mocking data.

19. Summary

A static dashboard overview using simulated data to showcase layout, theming, and intended KPIs.

Feature 2: Category Management

1. Feature Name

Category Management

2. Feature Purpose

Allows an LMS administrator to create, list, edit, and organize the categories under which courses are grouped. This helps structure the course catalog intuitively.

3. Feature Location

Frontend - Components: src/admin/pages/Categories/index.jsx, src/admin/pages/Categories/Detail.jsx, src/admin/pages/Categories/CreateCategory.jsx - Routes: src/routes/admin/categories/index.jsx, src/routes/admin/categories/$categoryId.jsx Backend - Controller: backend/course-service/src/main/java/com/xebia/lms/course/controller/CategoryController.java - Service: backend/course-service/src/main/java/com/xebia/lms/course/service/CategoryService.java - Entity: backend/course-service/src/main/java/com/xebia/lms/course/entity/Category.java

4. Related Files

Frontend - ✓ src/admin/pages/Categories/index.jsx - ✓ src/admin/pages/Categories/CreateCategory.jsx Backend - ✓ CategoryController.java - ✓ CategoryService.java - ✓ CategoryRepository.java - ✓ Category.java

5. Feature Entry Point

User navigates to `/categories` or clicks "New Category"

6. Complete Feature Lifecycle

User fills out the "Create Category" form. Submitting issues a POST /categories request to the Course Service. The Controller delegates to CategoryService to save the CategoryEntity via CategoryRepository. The UI then lists the newly created category in the grid view.

7. API Details

Method Endpoint Purpose
GET /categories Fetch all categories
POST /categories Create a new category
GET /categories/{id} Fetch a specific category
PUT /categories/{id} Update a specific category
DELETE /categories/{id} Delete a specific category

8. Database Flow

A single record is stored in the categories table with fields: id, name, description, icon, color, is_active.

9. State Management

Uses React Component state for managing the category list array and form loading states.

10. Component Hierarchy

<Categories Index>
  ├── <Search & Filter Bar>
  └── <Category Card Grid>
       └── <Category Card> (xN)

11. Sequence Diagram (Text)

1. Form submission triggered.
2. CategoryService.create() called on frontend.
3. POST /categories to API Gateway -> Course Service.
4. CategoryController -> CategoryService -> Repository.save().
5. Success response updates local React state array.

12. Dependencies

Spring Data JPA, Hibernate, PostgreSQL driver.

13. Validation Rules

CategoryRequest uses @Valid and @NotBlank on the name field to ensure the name is populated before attempting to save to the database.

14. Error Handling

Handled via Spring Boot's global exception handler which turns MethodArgumentNotValidException into HTTP 400 responses.

15. Security

No explicit `@PreAuthorize` tags; CORS is completely open (`@CrossOrigin(origins = "*", allowedHeaders = "*")`).

16. Important Classes and Methods

CategoryController.create()

17. Feature Dependencies

Used by Course Management to group courses.

18. Possible Improvements

Lock down CORS and add OAuth/JWT security.

19. Summary

Standard CRUD functionality for Categories allowing classification of the curriculum.

Feature 3: Course Management (Catalog & Admin Course CRUD)

1. Feature Name

Course Management (CRUD & Listing)

2. Feature Purpose

Empowers LMS administrators to view the complete catalog of courses, filter them by status or category, and edit existing courses. It acts as the gateway to the course authoring wizard and displays rich metadata like difficulty levels, thumbnails, publish status, and associated tags.

3. Feature Location

Frontend - Component: src/admin/pages/Courses/index.jsx (Course List Grid/List View) - Component: src/admin/pages/Courses/CreateCourse.jsx (Course Creation Wizard Step 1) - Component: src/admin/pages/Courses/Detail.jsx (Course Meta Detail Editing) Backend - Controller: backend/course-service/src/main/java/com/xebia/lms/course/controller/CourseController.java - Entity: backend/course-service/src/main/java/com/xebia/lms/course/entity/Course.java

4. Related Files

Frontend - ✓ src/admin/pages/Courses/index.jsx - ✓ src/admin/pages/Courses/CreateCourse.jsx - ✓ src/admin/pages/Courses/Detail.jsx - ✓ src/admin/services/api.js Backend - ✓ CourseController.java - ✓ CourseService.java - ✓ Course.java - ✓ CourseRequest.java

5. Feature Entry Point

User navigates to `/courses`

6. Complete Feature Lifecycle

Loads courses via CourseService.getCourses(). Lists courses. When "New Course" is clicked, user fills out title, description, and difficulty level. A POST request is made to /courses. Once created, the user is redirected to the Hierarchy Builder or Detail view.

7. API Details

Method Endpoint Purpose
GET /courses Fetch all courses
POST /courses Create a new course
PUT /courses/{courseId} Update a specific course
DELETE /courses/{courseId} Delete a specific course

8. Database Flow

Saved in courses table. Relates to categories via a nullable categoryId.

9. State Management

Local state for courses array and search/filter strings.

10. Component Hierarchy

<Courses Index>
  ├── <List / Grid Toggle>
  └── <Course List>

11. Sequence Diagram (Text)

1. User fills basic course details.
2. POST /courses with CourseRequestDTO.
3. Database inserts course.
4. Returns created course ID.
5. Router navigates to course detail/builder view.

12. Dependencies

JPA, Spring Web, Framer Motion (frontend list animations).

13. Validation Rules

Title and Description are enforced as non-blank fields.

14. Error Handling

Missing required fields return a 400 Bad Request which is surfaced via a toast notification on the UI.

15. Security

No active security.

16. Important Classes and Methods

CourseController.create(), Course.java

17. Feature Dependencies

Depends on Category availability.

18. Possible Improvements

Implement pagination (Pageable) instead of fetching all courses at once.

19. Summary

The core capability of creating the parent Course objects that will later hold content.

Feature 4: Course Authoring — Hierarchy Builder

1. Feature Name

Hierarchy Builder (Modules & Submodules)

2. Feature Purpose

Provides an interactive, intuitive drag-and-drop interface for structuring the educational curriculum of a course. Administrators can define ordered Modules (chapters) containing multiple ordered Submodules (lessons). This defines the skeleton of the course before adding actual learning content.

3. Feature Location

Frontend - Component: src/admin/pages/Courses/HierarchyBuilder.jsx Backend - Controller: CourseController.java (via /{courseId}/modules and /{courseId}/modules/{moduleId}/submodules)

4. Related Files

Frontend - ✓ src/admin/pages/Courses/HierarchyBuilder.jsx Backend - ✓ CourseController.java - ✓ CourseModule.java - ✓ SubModule.java - ✓ ModuleRequest.java - ✓ SubModuleRequest.java

5. Feature Entry Point

User accesses `HierarchyBuilder.jsx` after creating a Course.

6. Complete Feature Lifecycle

User clicks "Add Module". Form submits `ModuleRequest`. Next, under the module, user clicks "Add Submodule". Form submits `SubModuleRequest`.

7. API Details

Method Endpoint Purpose
POST /courses/{courseId}/modules Add a module to a course
POST /courses/{courseId}/modules/{moduleId}/submodules Add a submodule to a module
GET /courses/{courseId}/hierarchy Retrieve the full course structure

8. Database Flow

Saved in course_modules and submodules tables, maintaining position integer mapping.

9. State Management

Local state manages dragging arrays and opened/closed nested states.

10. Component Hierarchy

<HierarchyBuilder>
  ├── <Module Item>
       └── <Submodule Item>

11. Sequence Diagram (Text)

1. POST /courses/1/modules -> Creates Module
2. POST /courses/1/modules/1/submodules -> Creates Submodule
3. Updates position order integers on drag end.

12. Dependencies

None strictly beyond React standard libraries.

13. Validation Rules

Requires non-empty titles.

14. Error Handling

Basic error logging and UI toasts.

15. Security

Open access.

16. Important Classes and Methods

CourseController.module(), CourseController.subModule()

17. Feature Dependencies

Requires an existing Course entity.

18. Possible Improvements

Implement proper React dnd (drag and drop) library for improved cross-browser stability.

19. Summary

Builds the relational structure connecting Courses to Modules to Submodules.

Feature 5: Course Authoring — Content Manager

1. Feature Name

Content Manager

2. Feature Purpose

The core authoring tool that manages the rich content blocks attached to a specific Submodule. It supports various blocks such as Headings, Text, Code snippets, embedded Videos, Images, PDFs, Callouts, and Comparison Tables.

3. Feature Location

Frontend - Component: src/admin/pages/Courses/ContentManager.jsx Backend - Controller: CourseController.java (via /{courseId}/content-items) - Entity: ContentItem.java

4. Related Files

Frontend - ✓ src/admin/pages/Courses/ContentManager.jsx Backend - ✓ CourseController.java - ✓ ContentItem.java - ✓ ContentType.java - ✓ ContentItemRequest.java

5. Feature Entry Point

Access specific submodule edit mode.

6. Complete Feature Lifecycle

Admin selects a block type (e.g., Text). Admin types content. ContentItem object created with `type=NOTE`, `storageRef=rawText`, `courseId=X`, `subModuleId=Y`.

7. API Details

Method Endpoint Purpose
POST /courses/{courseId}/content-items Add a content item to a submodule/course

8. Database Flow

Inserts into content_items. type maps to the ContentType Enum (NOTE, VIDEO_REFERENCE, PDF, SCORM, INTERACTIVE_WIDGET, COMPARISON_TABLE, FLASHCARD).

9. State Management

Component state array holds current active blocks.

10. Component Hierarchy

<ContentManager>
  └── <Block Array>

11. Sequence Diagram (Text)

1. User creates text block.
2. Form posts ContentItemRequest.
3. ContentItem saved referencing course/submodule IDs.

12. Dependencies

None special.

13. Validation Rules

Type and CourseID must not be null.

14. Error Handling

400 Bad Request if missing IDs or Type.

15. Security

Open access.

16. Important Classes and Methods

CourseController.content()

17. Feature Dependencies

Requires existing Course/Module/Submodule structure.

18. Possible Improvements

Actual binary file storage endpoints (currently uses mocked or cloud links).

19. Summary

The lowest level of the hierarchy, handling actual instructional data types.
